# Repository Guidelines

A collection of independent, Windows-only C#/WPF **live-memory trainers** for classic DOS games running under **DOSBox / DOSBox-X**. Each trainer attaches to the running emulator process, signature-scans the emulated guest RAM to locate game state at runtime (addresses are never hard-coded), and reads/writes it live — with freeze toggles, "max" buttons, and, for some titles, offline save editing. Several also carry a reverse-engineering workspace behind their offsets.

## Project Structure & Module Organization

Nearly every top-level folder is a **self-contained trainer** with its own solution/project, `README.md`, a PowerShell run script (usually `Run.ps1`), and (mostly) its own `AGENTS.md` — read the per-project `AGENTS.md`/`README.md` before working inside one:

- `AutoduelTrainer/`, `BardsTale1Trainer/`, `BattleTech1Trainer/`, `ColonizationTrainer/`, `DarklandsTrainer/`, `DragonWarsTrainer/`, `KeefTrainer/`, `LordsOfTheRealmTrainer/`, `MightAndMagic1Trainer/`, `MinesOfTitanTrainer/`, `MoriaTrainer/`, `PoolOfRadianceTrainer/`, `QuestForGlory1Trainer/`, `ShogunTrainer/`, `SwordOfTheSamuraiTrainer/`, `SyndicatePlusTrainer/`, `ThePerfectGeneral2Trainer/`, `WarOfTheLanceTrainer/`, `WastelandTrainer/`.

`WastelandTrainer/` (Wasteland — Interplay/EA, 1988) is a full trainer: `src/WastelandTrainer/` (`net8.0-windows`, references `GameTrainers.Common` for both `Memory` and `Mvvm`), a `test/FormatCheck/` harness, its own `.sln`/`Run.ps1`/`README.md`/`AGENTS.md`, and Party/Inventory/References/Maps tabs. It locates the party by **structure** (an array of seven contiguous 256-byte records that pack from slot 0) rather than by a static anchor, and teleports by writing the party X/Y in the party-state header that precedes the roster. Its captured RAM dumps (`*.bin`, `*.csv`, `memdump.md`) live under `.data/` (git-ignored by `.*/`) — never `git add` the multi-hundred-MB dumps.

Layout varies: some use a single `Trainer/` project (no `.sln`), others `src/<Name>/` with a `.sln` and a `test/FormatCheck/` harness. `MightAndMagic1Trainer` is the architectural template the others were ported from. Common layers: Win32 P/Invoke (`Native*.cs`), process/guest-memory access, a game-knowledge layer holding all reverse-engineered offsets as constants, and hand-rolled MVVM (`ObservableObject`/`RelayCommand`).

`GameTrainers.Common/` is a shared library that extracts the game-agnostic plumbing out of that duplication: `GameTrainers.Common.Memory` (process/guest-memory access — `ProcessMemory`/`MemoryRegion`, `NativeMethods`, `MemorySearcher`, `MemoryDumper`, `DumpComparer`, `BytePatternScanner`, `GlobalHotkeys`, `KeyboardSender`) and `GameTrainers.Common.Mvvm` (`ObservableObject`/`RelayCommand`). The three MM1-family trainers (`MightAndMagic1Trainer`, `BardsTale1Trainer`, `PoolOfRadianceTrainer`) reference it instead of carrying their own copies; each keeps only its game-specific locators/scanners locally. `DragonWarsTrainer`, `MinesOfTitanTrainer`, `WarOfTheLanceTrainer`, and `WastelandTrainer` also reference it, pulling both `GameTrainers.Common.Memory` and `GameTrainers.Common.Mvvm` (via csproj `<Using>` items — note their `ObservableObject` uses `SetField`) and keeping only their own game-specific locators local (DragonWars its anchor-based `RosterLocator`, Mines of Titan its `IJKM`-anchored `PartyLocator`, War of the Lance its NAT.DAT + strength-block-signature `GameLocator`, Wasteland its structural seven-record `PartyLocator` plus a party-state-header teleport that has no separate locator — the header sits at a fixed offset before the located roster). `ThePerfectGeneral2Trainer` also references it, but is the odd one out: its target is a 16-bit DOS **protected-mode (DPMI)** program whose run-time heap has no stable static signature to anchor a locator to, so it has **no `GameLocator`** — instead it drives Common's `MemorySearcher` as a Cheat-Engine-style value scanner (attach → scan → narrow → pin/freeze) and keeps only its two Confirmed data structures (`PurchaseFormat`, `UnitReference`) local. `BattleTech1Trainer` follows the same value-scanner model for the same reason (its real-mode DOS target's mutable C-Bills/health/skill state has no adjacent constant byte-run to anchor a locator to, so it too has **no `GameLocator`**), but additionally uses Common's `BytePatternScanner` in a `GameDetector` **presence check** that signature-scans for the read-only ASCII tables `BTECH.EXE` maps into guest RAM verbatim; it keeps its Confirmed game-knowledge (`WeaponTable`, `WeaponReference`, `MechReference`, `SkillSheet`, `GameSignatures`) local. `PoolOfRadianceTrainer` intentionally kept its own divergent MVVM, `NativeMethods` and value-scanner and pulls only `ProcessMemory`/`MemoryRegion` from Common (via csproj using-aliases). `QuestForGlory1Trainer` also follows the value-scanner model: Quest for Glory I runs on Sierra's SCI0 interpreter whose heap is dynamically allocated in DOSBox guest RAM each session (no stable static signature), so it uses `MemorySearcher` for scanning; it additionally exposes a **Day/Time editor** (writes the game-clock tick counter and day global directly) and a **Teleport editor** (writes the current room-number global); it keeps only its game-knowledge layer (`GameOffsets`, `RoomBook`, `SkillBook`) local. `DarklandsTrainer` (Darklands — MicroProse, 1992) follows the same value-scanner model but is the most locked-down target: `DARKLAND.EXE` is a **PKLITE-compressed** MS-C binary that a DOS extender relocates into extended memory, so not only is the mutable state (attributes, skills, party Fame, the coin purse) dynamically allocated with no adjacent constant byte-run, but even the static tables are not at a fixed image offset and there is no reliable EXE-string detection signature — so it has **neither a `GameLocator` nor a `GameDetector`**, just `MemorySearcher` with five guided scans (Endurance/Strength/Skill/Fame/Florins). It keeps its Confirmed game-knowledge local (`AttributeBook` = six primaries + Divine Favor, `SkillBook` = nineteen skills, `GameFacts` = currency ratios + Fame ladder) plus a **read-only** `SaveFile` reader for the shipped `SAVES\DEFAULT` template (offsets derived from a single sample, so it never writes saves — `test/FormatCheck` validates it against a synthetic fixture). `MoriaTrainer` (The Dungeons of Moria / UMoria 5.5.2 — Koeneke/Wilson, 1988) follows the same value-scanner model: UMoria is a DJGPP v2.01-compiled 32-bit **DPMI** roguelike whose flat data segment is allocated at a session-specific linear address by CWSDPMI, so it has **neither a `GameLocator` nor a `GameDetector`**, just `MemorySearcher` with 14 guided scans (chp/maxhp/cmana/mhp/au/exp/lev/str/int/wis/dex/con/chr/food). It additionally exposes a **relative-scan teleport** that locates `char_row`/`char_col` (two `int16` globals that track cardinal moves) by snapshotting an unknown-value Int16 baseline, walking one square in-game, and narrowing by Increased/Decreased until two candidates remain — a workflow unique among the value-scanner trainers because UMoria is the only one with a single character that steps cardinal directions. It keeps its Confirmed game-knowledge local (`PlayerFormat` = the `player_type` struct offsets + the 3..18/100 stat encoding + the cave cell constants, `RaceBook`/`ClassBook`/`SpellBook`/`ItemBook`/`LevelBook`/`MonsterBook`/`ParagraphBook` = the eight races, six classes, 31+31 spells, 27 item categories, 51-level descent reference, a curated 29-creature roster including the Balrog, and the recall-paragraph renderer) — `test/FormatCheck` validates the stat encoding round-trips, the counts, and the scan helpers. `ColonizationTrainer` (Sid Meier's Colonization — MicroProse, 1994) also references Common but is architecturally distinct: its **primary** feature is an **offline save-game editor**, not a live trainer. The `COLONYxx.SAV` format is a flat little-endian serialization (signature `"COLONIZE"`, no checksum), so it edits fields in place through typed record views — the human player's **gold** is a 4-byte value at nation-record offset `0x2A`, whose absolute position is computed from the header's `colony_count`/`unit_count` (sections after the 390-byte header are count-addressed, never fixed), verified byte-for-byte against the shipped saves and three community sources that decoded `VICEROY.EXE`. It keeps its own game layer local (`SaveFormat` offset table, `Bytes` LE accessor, `SaveGame`/`NationRecord`/`ColonyRecord` views, and the goods/unit/terrain/profession/building/founding-father/nation reference books) and additionally exposes Common's `MemorySearcher` as a live value scanner for gold/tax/bells (the real-mode overlaid `VICEROY.EXE` has no session-stable signature to anchor a locator to, so there is **no `GameLocator`**); `test/FormatCheck` round-trips a synthetic save byte-for-byte and, when the copyrighted `.games\COLONY00.SAV` is present, asserts the empirically-confirmed values (map 58×72, year 1492, England, 0 gold/0% tax). The remaining trainers are still self-contained. So a fix to the shared plumbing lands once for all of them; a fix in one trainer's local copy of the old code still needs porting.

Dot-prefixed dirs (`.docs/` teardown & strategy notes, `.data/` RAM dumps, `.game/` copyrighted assets) are **git-ignored** (`.gitignore` `.*/`) — never commit them.

## Build, Test, and Development Commands

- Root launcher: from the repo root run `.\Run.ps1` for an interactive menu of every trainer, or `.\Run.ps1 -Trainer <name|number>` to pick one directly (`-List` prints the menu and exits). It discovers any top-level folder that contains its own `Run.ps1` and forwards the shared options below to it.
- Per-trainer: `.\Run.ps1` inside each trainer folder (including `KeefTrainer`, which now has one). Restores, builds Release, and launches; a UAC prompt appears because the app manifests request admin for `Read/WriteProcessMemory`.
- Every `Run.ps1` (root and per-trainer) exposes the **same** options: `-Configuration Debug|Release` (default Release), `-Clean` (remove bin/obj first), `-NoBuild` (skip build, launch the existing exe), `-NoRun` (build only), `-Test` (run the verification harness; warns if the trainer has none), and `-Publish` (single self-contained win-x64 exe, skips launch).
- Direct: `dotnet build <project>.csproj -c Release` / `dotnet run --project <test-project>`.

## Coding Style & Naming Conventions

C# targeting `net8.0-windows` or `net9.0-windows`, **x64**, with `Nullable` and `ImplicitUsings` enabled. No `.editorconfig`, linter, or formatter is committed — match the surrounding file: 4-space indent, file-scoped namespaces, `sealed` classes, PascalCase members, `_camelCase` fields, `const` hex for offsets, and XML `///` docs on public members. Keep reverse-engineered constants in the game layer; follow the read-validate-write pattern so a shifted layout is never corrupted.

## Testing Guidelines

There is no xUnit/NUnit suite. Trainers that have one use a headless console harness — usually `FormatCheck` (`net8.0` or `net8.0-windows`), or `Verify` in `SwordOfTheSamuraiTrainer` — that asserts parsers against captured dumps/save files and exits 0 (pass) or 1 (fail). The GUI cannot be smoke-tested headlessly — it needs an interactive desktop and a running game. Keep the harness green.

## Commit & Pull Request Guidelines

The root repository has no commit history yet. Per the sibling projects, write imperative, sentence-case subjects (join related changes with a semicolon); describe which game state a change reads/writes and how it was confirmed against the live game.
